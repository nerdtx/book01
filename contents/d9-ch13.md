## Section 9: Handle the Player’s Turn Through Event Handlers

At this point, the app has:

* a stable HTML shell
* styled boards
* a centralized state object
* rendering functions
* ship placement logic
* hit detection logic

Now it is time to connect the game to the player.

In a single page app, this usually happens through **event handlers**. An event
handler listens for something the user does and responds by updating the app’s
state. In this game, the most important user action is clicking a square on the
**computer board**.

This section focuses only on the **player’s turn**. The computer’s turn will come next.

### The Basic SPA Flow

A well-structured event handler should follow a consistent pattern:

1. Check whether the action is allowed
2. Update the state
3. Update any messages
4. Re-render the app
5. Save the state if needed

That pattern keeps the app predictable and easier to debug. (Are you sensing a
pattern? Predictable and easy to debug is Good.)

### Only Allow Shots During the Player’s Turn

The player should only be able to fire when:

* the game is still being played
* it is currently the player’s turn
* the chosen square has not already been targeted

That means the event handler needs to guard against invalid actions before
changing anything.

### Write the Player Shot Handler

Create this function:

```js id="n4x8p2"
function handlePlayerShot(row, col) {
  if (gameState.status !== "playing") {
    return;
  }

  if (gameState.turn !== "player") {
    return;
  }

  if (gameState.playerShots[row][col] !== "") {
    return;
  }

  if (computerHasShipAt(row, col)) {
    gameState.playerShots[row][col] = "hit";
    gameState.playerHits++;
    renderMessage("Hit!");
  } else {
    gameState.playerShots[row][col] = "miss";
    gameState.playerMisses++;
    renderMessage("Miss!");
  }

  checkForGameEnd();

  if (gameState.status === "playing") {
    gameState.turn = "computer";
  }

  saveGame();
  renderApp();
}
```

This function does several important things, but each step is still clear.

### Understanding the Guard Clauses

The first three `if` statements are called **guard clauses**. They stop the
function early if the action should not happen.

```js id="e7q1m6"
if (gameState.status !== "playing") {
  return;
}
```

This prevents clicks after the game has already ended.

```js id="p6v3r8"
if (gameState.turn !== "player") {
  return;
}
```

This prevents the player from firing during the computer’s turn.

```js id="j9k5w4"
if (gameState.playerShots[row][col] !== "") {
  return;
}
```

This prevents duplicate shots on the same square.

These early returns make the rest of the function easier to read because invalid
cases are removed immediately.

> **Helpful Hint:**
> Guard clauses are a great way to simplify event handlers. They let you reject
> bad actions early and keep the main logic cleaner.

### Update the Shot Grid, Not the Board Directly

Notice that the function does **not** directly color a cell or insert text into
the board. Instead, it updates the state:

```js id="z8u2c7"
gameState.playerShots[row][col] = "hit";
```

or:

```js id="r3m7d1"
gameState.playerShots[row][col] = "miss";
```

That is an important SPA habit. The event handler changes the state, and the
render functions later update the DOM.

This keeps the app consistent. If the page ever needs to be rebuilt, the shot
grids still contain the truth.

### Update the Score Counters

The function also updates the score values inside the state object:

```js id="h5n4x9"
gameState.playerHits++;
```

or:

```js id="w2p6k3"
gameState.playerMisses++;
```

Again, the state changes first. The scoreboard on the page will update when
`renderScoreboard()` runs inside `renderApp()`.

### Check for a Win After the Shot

After recording the shot, the game should check whether the player has sunk all
of the computer’s ships.

For now, the handler calls a placeholder function:

```js id="u7r9b5"
checkForGameEnd();
```

You will build that logic in a later section. For now, this keeps the structure
clean and prepares the event flow.

### Switch Turns Only If the Game Continues

If the shot did not end the game, control passes to the computer:

```js id="s4q1v8"
if (gameState.status === "playing") {
  gameState.turn = "computer";
}
```

This is important because the app should not hand the turn to the computer after
the player has already won.

### Save and Re-Render

At the end of the event handler, the app saves the new state and redraws the page:

```js id="d6m2k7"
saveGame();
renderApp();
```

This completes the standard single page app cycle:

* user action
* state update
* app render

### Connect the Handler to the Computer Board

In Section 5, the `renderComputerBoard()` function attached click handlers to
each cell:

```js id="k8w3n1"
cell.addEventListener("click", function () {
  handlePlayerShot(row, col);
});
```

That means the board is already wired to use this new function. Once
`handlePlayerShot()` exists, clicking a computer-board cell will trigger the
correct behavior.

### Why This Is Good SPA Structure

This event handler is a strong example of organized single page app design
because it:

* validates the action first
* updates only the state
* avoids direct board manipulation
* uses render functions instead of manual DOM patching
* stays focused on one user action

It does not try to handle the computer’s turn yet. That is a separate
responsibility and belongs in its own section.

Keeping those concerns separate will make the game easier to extend and debug.

* update the score
