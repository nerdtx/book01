## Section 8: Detect Hits Without Touching the DOM

Now that the app can initialize itself correctly, the next step is to write the
core game logic for hits and misses. This is one of the most important moments
in the chapter, because it reinforces a major single page app principle:

**Game logic should not depend on the DOM.**

In other words, the code that decides whether a shot is a hit should not care
about colors, text labels, CSS classes, or what the page looks like. It should
only work with data.

That separation makes the app easier to test, easier to debug, and easier to
extend.  It makes the whole thing easier to write.

### Keep Game Rules Separate from Display

Suppose the player clicks a square on the computer board. The app needs to
answer a very specific question:

> Does this square contain part of a ship?

That is a logic question, not a display question.

A poor design would mix this answer directly into rendering code. A better
design creates small helper functions that work entirely from the state object.

### Write a General Ship Lookup Function

Instead of writing one function for player ships and another for computer ships,
create one reusable helper that checks whether a fleet contains a ship at a
given position.

```js
function fleetHasShipAt(row, col, fleet) {
  for (let ship of fleet) {
    for (let cell of ship.cells) {
      if (cell.row === row && cell.col === col) {
        return true;
      }
    }
  }

  return false;
}
```

This function takes:

* a row
* a column
* a fleet

It returns:

* `true` if any ship in that fleet occupies that square
* `false` otherwise

Because the function is generic, it works for both sides of the game.

### Use It for the Player and the Computer

Now you can create small wrappers that make your code easier to read later.

```js
function playerHasShipAt(row, col) {
  return fleetHasShipAt(row, col, gameState.playerShips);
}

function computerHasShipAt(row, col) {
  return fleetHasShipAt(row, col, gameState.computerShips);
}
```

These helper functions are simple, but they improve readability throughout the rest of the app.

For example:

* `playerHasShipAt(2, 3)` is easier to understand than
* `fleetHasShipAt(2, 3, gameState.playerShips)`

### Why This Structure Is Useful

These functions are pure logic helpers. They do not:

* update the page
* add CSS classes
* change the score
* show messages
* save the game

They just answer a question.

That is exactly what you want in a well-structured single page app. Each
function should do one thing clearly.

> **Helpful Hint:**
> If a function both checks a rule and updates the screen, it is probably doing
> too much.

### Detecting Hits Is Not the Same as Recording Shots

It is important to distinguish between two ideas:

* **Does a ship exist at this square?**
* **Has this square already been fired on?**

The first question is answered by the ship-checking functions you just wrote.

The second question is answered by the shot grids:

* `gameState.playerShots`
* `gameState.computerShots`

Those grids keep track of previous attacks. The fleet data keeps track of ship
locations. These are two different pieces of information, and keeping them
separate makes the app cleaner.

### Example: Understanding the Difference

Suppose this is part of the computer’s fleet:

```js
gameState.computerShips = [
  {
    cells: [
      { row: 1, col: 1 },
      { row: 1, col: 2 }
    ]
  }
];
```

And the player’s shots grid contains:

```js
gameState.playerShots = [
  ["", "", "", "", "", ""],
  ["", "hit", "", "", "", ""],
  ["", "", "", "", "", ""],
  ["", "", "", "", "", ""],
  ["", "", "", "", "", ""],
  ["", "", "", "", "", ""]
];
```

From this, the app can tell:

* the computer has a ship at `(1, 1)`
* the player has already hit `(1, 1)`
* the computer also has a ship at `(1, 2)`
* the player has not yet fired at `(1, 2)`

This is why the state needs both fleets and shot grids.

### Build a Reusable Hit Test Function

You may also want one helper that clearly returns whether a shot should count as a hit against a fleet.

```js
function isHitOnFleet(row, col, fleet) {
  return fleetHasShipAt(row, col, fleet);
}
```

This may look simple, but it gives you a very readable way to express game rules later.

For example:

```js
if (isHitOnFleet(row, col, gameState.computerShips)) {
  console.log("Player scored a hit.");
}
```

Readable code matters, especially in a larger project.

### Prepare for the Next Step

At this point, your app can now answer the most important game-rule question for
either side:

* Did the player hit the computer?
* Did the computer hit the player?

That logic is ready to be used, but nothing in the app responds to clicks yet.
The hit detection is separate and reusable, which is exactly what you want
before connecting it to events.

turn. That section will connect clicks on the computer board to state updates,
