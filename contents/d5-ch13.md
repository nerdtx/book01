## Section 5: Separate Rendering from Logic

Now that the app has a centralized state object, the next important step is to
keep **rendering** separate from **game logic**.

This is one of the most important structural ideas in a single page app.

A common beginner mistake is to write one large function that does everything at once:

* changes the state
* updates the board
* updates the score
* changes the message
* checks for a win
* saves to `localStorage`

That approach may seem convenient at first, but it quickly becomes difficult to
read and maintain. Instead, a better design separates responsibilities:

* **Game logic functions** change the state
* **Render functions** update the page
* **Event handlers** connect user actions to the logic

This makes the app easier to debug because each function has a clear purpose.

### What Rendering Means

A **render function** reads from the current state and updates the page so that
the user sees the correct information.

Rendering should not decide the rules of the game. It should not place ships or
determine whether a shot is a hit. Its job is only to display what the state
already says is true.

For example:

* if `gameState.playerHits` is `3`, the scoreboard should show `3`
* if `gameState.turn` is `"computer"`, the message area should reflect that
* if `gameState.playerShots[2][4]` is `"miss"`, the matching computer-board cell should look like a miss

### Start with Small Render Functions

Instead of one giant render function, begin with several smaller ones.

#### Render the Message Area

```js
function renderMessage(text) {
  document.getElementById("message").textContent = text;
}
```

This function updates the status message shown to the player.

#### Render the Scoreboard

```js
function renderScoreboard() {
  document.getElementById("playerHits").textContent = gameState.playerHits;
  document.getElementById("playerMisses").textContent = gameState.playerMisses;
  document.getElementById("computerHits").textContent = gameState.computerHits;
  document.getElementById("computerMisses").textContent = gameState.computerMisses;
}
```

This function updates the four visible score values from the state object.

> **Helpful Hint:**
> Good render functions do not guess. They simply read the current state and display it.

### Render the Player Board

The player board needs to show two kinds of information:

* where the player’s ships are
* where the computer has fired

That means the board must reflect both ship placement and shot results.

A simple strategy is:

* always show player ships
* mark hits and misses from `computerShots`

Example:

```js
function renderPlayerBoard() {
  const board = document.getElementById("playerBoard");
  board.innerHTML = "";

  for (let row = 0; row < gameState.size; row++) {
    for (let col = 0; col < gameState.size; col++) {
      const cell = document.createElement("div");
      cell.classList.add("cell");

      if (playerHasShipAt(row, col)) {
        cell.classList.add("ship");
      }

      const shotState = gameState.computerShots[row][col];

      if (shotState === "hit") {
        cell.classList.add("hit");
        cell.textContent = "X";
      } else if (shotState === "miss") {
        cell.classList.add("miss");
        cell.textContent = "O";
      }

      board.appendChild(cell);
    }
  }
}
```

This function uses another helper, `playerHasShipAt(row, col)`, which you will
write shortly.

### Render the Computer Board

The computer board should behave differently:

* do **not** show ship locations
* show the player’s hits and misses

Example:

```js
function renderComputerBoard() {
  const board = document.getElementById("computerBoard");
  board.innerHTML = "";

  for (let row = 0; row < gameState.size; row++) {
    for (let col = 0; col < gameState.size; col++) {
      const cell = document.createElement("div");
      cell.classList.add("cell");

      const shotState = gameState.playerShots[row][col];

      if (shotState === "hit") {
        cell.classList.add("hit");
        cell.textContent = "X";
      } else if (shotState === "miss") {
        cell.classList.add("miss");
        cell.textContent = "O";
      }

      cell.addEventListener("click", function () {
        handlePlayerShot(row, col);
      });

      board.appendChild(cell);
    }
  }
}
```

This function adds click behavior to each computer-board cell so the player can fire at it.

### Write a Ship Lookup Helper

To support rendering the player board, write a helper function that checks
whether a ship occupies a given cell.

```js
function playerHasShipAt(row, col) {
  for (let ship of gameState.playerShips) {
    for (let cell of ship.cells) {
      if (cell.row === row && cell.col === col) {
        return true;
      }
    }
  }

  return false;
}
```

This function is a good example of separation of concerns. It answers one clear
question and does not touch the DOM.

### Build a Top-Level Render Function

Once you have several smaller render functions, it helps to create one master render function that updates the whole app.

```js
function renderApp() {
  renderScoreboard();
  renderPlayerBoard();
  renderComputerBoard();
}
```

Now, whenever the state changes, you can call `renderApp()` to refresh the interface.

That creates a strong SPA pattern:

1. state changes
2. render functions run
3. interface matches the state

### Keep Rendering Predictable

Try to keep rendering simple and repeatable. A render function should not depend
on hidden side effects or half-updated values. It should always build the page
from the current state.

This predictability makes the app much easier to test and reason about.

### Structure So Far

At this point, the app already has a better structure than many beginner projects:

* the HTML provides a stable shell
* the CSS defines how app regions look
* the state object stores the game’s truth
* render functions update the screen from the state

That is exactly what you want in a single page app.
