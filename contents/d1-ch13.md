## Section 1: Plan the App Before the Game

In this chapter, you will build a simple Battleship game as a **single page app**.
This means the project has two goals at the same time:

* build a game that works
* build it with clear, organized app structure

That second goal matters just as much as the first one.

A messy program may still function, but it becomes difficult to debug, extend,
or improve. A well-structured single page app keeps its responsibilities
separated and its state easy to understand. This chapter will emphasize that
structure from the very beginning.

### The Game Rules

This version of Battleship is a **single-player game against the computer**.

The rules are:

* There are **two boards**

  * the **player board**
  * the **computer board**
* Both the player and the computer have ships
* Ships are placed randomly
* Ships may be placed horizontally or vertically
* Ships may not overlap
* The player clicks squares on the **computer board** to fire
* After the player fires, the computer randomly fires at the **player board**
* A shot is either a hit or a miss
* The game ends when one side has all ship cells hit

This version is still simplified, but it is much closer to a real Battleship
game because both sides take turns.

### Think Like an App Designer

Before writing code, step back and describe what the app must do.

This project is not just a board and some clicks. It is a single page app with
several responsibilities:

* **State** - what the app must remember
* **Rendering** - what the app must display
* **Game Logic** - the rules of Battleship
* **Event Handling** - how the app responds to clicks
* **Persistence** - how the app saves progress

If these responsibilities are mixed together carelessly, the code becomes
confusing. For example, if one click handler tries to:

* check for a hit
* update the score
* decide whether the game is over
* update both boards
* save the game
* choose the computer’s move

then that one part of the program becomes too difficult to manage.

A better design uses small functions with clear purposes.

### The State of the App

In a single page app, the **state** is the source of truth. It stores what the
app currently knows.

For this Battleship game, the app will eventually need to remember:

* the board size
* the player’s ships
* the computer’s ships
* the player’s shots against the computer
* the computer’s shots against the player
* the number of hits and misses on each side
* whose turn it is
* whether the game is still being played, won, or lost

That is a lot of information, which is exactly why structure matters.

A good single page app should be able to rebuild its interface just from the
state. If the state is accurate, the screen can be redrawn correctly.

### The Interface

The page itself will need several major regions:

* a title
* a status message
* a scoreboard
* a **player board**
* a **computer board**
* a control area with a New Game button

These parts form the app shell. The shell stays in place while JavaScript
updates what the user sees.

### The Turn Cycle

Because this is a two-sided game, the app needs to manage turns carefully.

A normal cycle will look like this:

1. The player clicks a square on the computer board
2. The app checks for a hit or miss
3. The state updates
4. The computer takes a turn
5. The app checks for a hit or miss on the player board
6. The state updates again
7. The interface re-renders
8. The app checks for a win or loss

This is one of the biggest differences between a simple click-based project and
a more complete single page app. The app is not only reacting to the player. It
is also managing an opponent’s behavior and a turn-based flow.

> **Helpful Hint:**
> When planning a single page app, ask two questions over and over: “What
> changed in the state?” and “What should the page look like now?” Those two
> questions keep your code organized.

### What This Chapter Is Really Teaching

On the surface, this chapter is about Battleship. Underneath, it is about how to
design a small but complete single page app.

The Battleship game gives you a reason to practice:

* centralized state
* multiple views of data
* clear rendering functions
* turn-based logic
* persistent storage
* organized event handling

That makes this chapter a capstone project. You are not just making a game. You
are building an app with moving parts that must stay synchronized.
