## Section 4: Create a Centralized State Object

In a single page app, the most important rule is this:

**The state object is the source of truth.**

That means the app should not depend on scattered variables, hidden assumptions,
or whatever happens to be on the screen. Instead, the real condition of the game
should live in one organized data structure.

If the state is correct, the interface can be rebuilt from it.

This matters even more in a game like Battleship, because the app must remember
many pieces of information at once:

* the size of the board
* where the player’s ships are
* where the computer’s ships are
* which shots the player has fired
* which shots the computer has fired
* the current turn
* hit and miss counts
* whether the game is still being played

That is exactly the kind of situation where a centralized state object becomes valuable.

### Why a State Object Helps

Without a clear state object, you might end up with variables like these spread
throughout your file:

```js
let playerHits = 0;
let computerHits = 0;
let boardSize = 6;
let playerTurn = true;
let playerShips = [];
let computerShips = [];
let playerShots = [];
let computerShots = [];
```

This can work for a while, but it becomes harder to manage as the project grows.
A state object groups related data together and makes the app easier to reason
about.

A better design looks like this:

```js
let gameState = {
  size: 6,
  playerShips: [],
  computerShips: [],
  playerShots: [],
  computerShots: [],
  playerHits: 0,
  playerMisses: 0,
  computerHits: 0,
  computerMisses: 0,
  turn: "player",
  status: "playing"
};
```

This object gives the app one central place to check what is happening.

### What Each Part Means

Here is what each property stores:

* `size` - the number of rows and columns on each board
* `playerShips` - the player’s fleet
* `computerShips` - the computer’s fleet
* `playerShots` - the player’s attacks against the computer board
* `computerShots` - the computer’s attacks against the player board
* `playerHits` - how many successful shots the player has made
* `playerMisses` - how many misses the player has made
* `computerHits` - how many successful shots the computer has made
* `computerMisses` - how many misses the computer has made
* `turn` - whose turn it currently is
* `status` - whether the game is still active or already finished

This structure is much easier to understand than a loose collection of unrelated variables.

### Representing Shot Grids

The game needs a way to remember which cells have already been targeted. The
simplest approach is to use a two-dimensional array for each shot grid.

Each cell in the shot grid can contain:

* `""` for untouched
* `"hit"` for a hit
* `"miss"` for a miss

That means both `playerShots` and `computerShots` will eventually be 6 by 6 grids.

Here is a helper function that creates an empty grid:

```js
function createEmptyGrid(size) {
  let grid = [];

  for (let row = 0; row < size; row++) {
    let rowData = [];

    for (let col = 0; col < size; col++) {
      rowData.push("");
    }

    grid.push(rowData);
  }

  return grid;
}
```

You can use it like this:

```js
gameState.playerShots = createEmptyGrid(gameState.size);
gameState.computerShots = createEmptyGrid(gameState.size);
```

Now both shot grids begin in a clean, predictable state.

### Representing Ships

Each ship should be stored as an object with a list of cells. That list will
tell the app exactly where the ship occupies space on the board.

Example:

```js
{
  cells: [
    { row: 1, col: 2 },
    { row: 1, col: 3 },
    { row: 1, col: 4 }
  ]
}
```

Then a fleet becomes an array of ship objects:

```js
let exampleFleet = [
  {
    cells: [
      { row: 0, col: 0 },
      { row: 0, col: 1 }
    ]
  },
  {
    cells: [
      { row: 3, col: 2 },
      { row: 4, col: 2 },
      { row: 5, col: 2 }
    ]
  }
];
```

This design keeps the ship data separate from the shot data, which is a good
thing. The ships describe where the fleet is. The shot grids describe what
attacks have happened.

### Turn and Status

The `turn` property helps control who acts next. In this game, it should usually be either:

* `"player"`
* `"computer"`

The `status` property describes the larger state of the game. For now, the app
can use values like:

* `"playing"`
* `"playerWon"`
* `"computerWon"`

This is better than using vague values like `true` or `false`, because the
meaning is clearer.

> **Helpful Hint:**
> When a variable or property can mean several different things, use descriptive
> string values instead of forcing everything into `true` and `false`.

### Initialize the State Properly

To avoid repeating setup logic, create a function that builds a fresh state object:

```js
function createNewGameState() {
  return {
    size: 6,
    playerShips: [],
    computerShips: [],
    playerShots: createEmptyGrid(6),
    computerShots: createEmptyGrid(6),
    playerHits: 0,
    playerMisses: 0,
    computerHits: 0,
    computerMisses: 0,
    turn: "player",
    status: "playing"
  };
}
```

Then initialize the app like this:

```js
let gameState = createNewGameState();
```

This makes resetting the game easier later, because you can call the same
function again.

### Why This Matters for a Single Page App

A well-structured single page app does not depend on the page itself to remember
what is happening. The DOM is for display. The state object is for truth.

That distinction is critical.

If you ever find yourself asking, “What is the current score?” or “Whose turn is
it?” the answer should come from the state object, not from reading text off the
page.

That is one of the most important habits in frontend programming.

functions that update the page from the current state.
