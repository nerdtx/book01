## Section 7: Create a Proper App Initialization Flow

Single page apps benefit from a clear startup process. Instead of scattering
setup code throughout the file, a well-structured app should have one obvious
path for starting a new game and one obvious path for loading an existing game.

This section builds that flow.

The Battleship app must be able to:

* create a fresh state
* place ships for both sides
* reset all shot grids
* reset all scores
* set the correct starting turn
* save the game if needed
* render the app

That is a lot of work, which is exactly why it should be organized carefully.

### Why Initialization Matters

In a small script, it may seem acceptable to place setup code directly at the
bottom of the file:

```js
gameState.playerShips = placeFleet(gameState.size);
gameState.computerShips = placeFleet(gameState.size);
gameState.playerShots = createEmptyGrid(gameState.size);
gameState.computerShots = createEmptyGrid(gameState.size);
renderApp();
```

But that becomes harder to manage later when you need to:

* start a new game from a button
* restore a saved game
* test different states
* reset the app after a win or loss

A better design uses dedicated functions so that startup behavior is predictable
and reusable.

### Create a Fresh State Function

You already created a helper that returns a blank game state. Now expand that
into a complete game setup flow.

Remember this from a few sections ago?

```js
function createNewGameState() {
  return {
    size: 6,
    playerShips: [],
    computerShips: [],
    playerShots: createEmptyGrid(6),
    computerShots: createEmptyGrid(6),
    playerHits: 0,
    playerMisses: 0,
    computerHits: 0,
    computerMisses: 0,
    turn: "player",
    status: "playing"
  };
}
```

This creates the structure, but the fleets are still empty.

### Add Fleet Placement During Setup

Now create a function that uses that blank state and finishes the setup.

```js
function setupNewGame() {
  gameState = createNewGameState();

  gameState.playerShips = placeFleet(gameState.size);
  gameState.computerShips = placeFleet(gameState.size);
}
```

This function creates a fully populated game state for a fresh game.

Notice that it does not render the page yet. That is deliberate. It is still
focused on state preparation.

### Create a New Game Function

Now build a function that handles everything required when the player starts over.

```js
function newGame() {
  setupNewGame();
  saveGame();
  renderMessage("Click a square on the computer board to fire.");
  renderApp();
}
```

This is a very important function because it ties several app responsibilities
together in a clean way:

1. prepare state
2. save it
3. update the message
4. render the page

Each of those jobs still uses its own helper function, which keeps the code
readable.

> **Helpful Hint:**
> A strong initializer does not do all the work itself. It coordinates other
> focused functions.

### Connect the New Game Button

Now make the New Game button actually work.

```js
document.getElementById("newGameBtn").addEventListener("click", newGame);
```

This means that every time the player clicks the button, the app will reset to a
clean state and redraw the interface.

### Create an App Startup Function

At this point, the app also needs a clear entry point when the page first loads.
In a single page app, that is usually handled by one startup function.

```js
function initApp() {
  document.getElementById("newGameBtn").addEventListener("click", newGame);
  loadGame();
}
```

This function does two things:

* attaches the New Game button event handler
* tries to load an existing saved game

If no saved game exists, `loadGame()` will eventually fall back to creating a
new one.

This gives the app one clean place where startup begins.

### Start the App

At the bottom of your script file, call the initializer:

```js
initApp();
```

That line should appear only once, and it should be easy to find.

### Why This Is Good SPA Structure

This initialization flow shows good single page app design because it separates
startup into clear stages:

* `createNewGameState()` builds the base state object
* `setupNewGame()` fills in the fleets
* `newGame()` coordinates state creation, saving, and rendering
* `initApp()` starts the application

Each function has a clear responsibility, and together they create a reliable
startup path.

If you later add new features, such as difficulty levels or manual ship
placement, you will know exactly where that setup logic belongs.

### Avoiding Common Problems

Without a clear initialization flow, you may run into issues like:

* rendering before the fleets exist
* attaching event listeners multiple times
* forgetting to reset part of the state
* loading a saved game incorrectly
* duplicating startup logic in multiple places

Good structure prevents these problems before they appear.

### What the App Can Do Now

At this point, your Battleship app can:

* create a clean starting state
* randomly place ships for both sides
* reset itself with a button
* start from a known initialization flow

That is a major step forward.
